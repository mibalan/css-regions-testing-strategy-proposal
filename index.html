This document is intended to be used as a guideline for the testing activities related to the CSS Regions spec [[!css3-regions]]. Its main goal is to provide an overview of the general testing areas, possible caveats and testing aspects not immediately apparent from the spec. Also, it provides a means of tracking the progress of the CSS Regions spec testing. This document is not meant to replace the spec in determining the normative and non-normative assertions to be tested, but rather complement it.

Introduction

- specs are getting more complex, testing efforts need coordination, esp with things like TtWF, where people less familiar with the whole W3C status/policy/relationships want to contribute - complement the existing test suite - this is what should be done, at a minimum, the test suite is what has already been done - general approach to testing regions, the goals for this testing, testing areas (normative & cross referenced) - notable people to refer to in regards to testing css regions (me, Alan, Rebecca, others?) - other supporting documents (??) - test progress tracking - based on existing W3C tests and

Approach

- testing the spec cannot be entirely separated from testing an implementation - it's a complex spec, so multiple passes will be needed - each going deeper in some aspects - general points to keep in mind - at each pass, get most bang for the buck - try to be as automation friendly as possible - ref-tests/testharness.js tests - keep manual tests to a minimum - where manual tests are absolutely required, try to design them with possible future automation in mind; there are efforts to enable testing with WebDriver [[webdriver]]

Goals

- be as automation-friendly as possible - side-goal is to play nicely with implementors automation tools/suites, so that they can contribute as much as possible to the W3C testing effort - see above about ref-tests/testharness.js over manual tests - provide guidance on areas' testing needs (what needs how much testing, caveats, etc.) - enable contributions from the community (other implementors, non-affiliated ppl) - "layered" testing, spec tests should be usable to get an idea of the level of support in a certain implementation

Risks and mitigation

Risks

- implicit interactions with a lot of other spec - some are in CSSWG (flexbox, grid, etc.) - some are in "fluid" specs (e.h. HTML5) or non-CSSWG (ShadowDOM) - (supposed) big number of tests needed - TODO ask Rebecca about spec coverage reports

Mitigation

- for mature specs, try to make the two play nice together, for less mature specs, try to be future proof, raise concerns, but don't try to cater for everything (must draw a line) - see goal about test coordination and enabling easy test contribution
